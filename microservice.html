Microservices Architecture

Clean Architecture & Onion Pattern - A Practical Guide

Introduction

Clean Architecture and Onion Pattern are architectural approaches that help build maintainable, testable, and scalable microservices. These patterns separate concerns and create clear boundaries between different layers of an application, making it easier to evolve and maintain over time.

This guide explains both patterns with practical examples and diagrams, drawing from real-world implementations and best practices.

Clean Architecture

Clean Architecture, introduced by Robert C. Martin (Uncle Bob), is an architectural pattern that emphasizes separation of concerns and independence of frameworks, UI, and databases. The core principle is that business logic should be independent of external concerns.

Core Principles

  • Independence of Frameworks: Business logic doesn't depend on external libraries
  • Testability: Business logic can be tested without UI, database, or external services
  • Independence of UI: UI can be changed without affecting business logic
  • Independence of Database: Business logic is not tied to a specific database
  • Independence of External Services: Business rules don't depend on external APIs
Clean Architecture Layers

Layer Responsibilities

1. Entities (Core)

Enterprise-wide business rules. These are the core business objects that encapsulate the most general and high-level rules. They are independent of any framework or technology.

2. Use Cases (Application Layer)

Application-specific business rules. These orchestrate the flow of data to and from entities and direct those entities to use their enterprise-wide business rules to achieve the goals of the use case.

3. Interface Adapters

Convert data between the format most convenient for use cases and entities, and the format most convenient for external agencies (like databases, web, etc.). This layer contains controllers, presenters, and gateways.

4. Frameworks & Drivers

The outermost layer containing frameworks and tools. This includes web frameworks, database tools, and external service integrations. This layer is where all the details go.

Onion Architecture

Onion Architecture, introduced by Jeffrey Palermo, is similar to Clean Architecture but emphasizes the dependency rule more explicitly. The name comes from the layered structure that resembles an onion, with the core at the center and dependencies pointing inward.

Onion Architecture Layers

Key Characteristics

  • Dependency Inversion: Dependencies point inward toward the domain core
  • Independence: Core domain is independent of infrastructure and UI
  • Testability: Core can be tested without external dependencies
  • Flexibility: Easy to swap out infrastructure components
  • Focus on Domain: Business logic is at the center

Clean Architecture vs Onion Architecture

Both patterns share similar goals but have slight differences in terminology and emphasis:

Similarities:

  • Both emphasize dependency inversion
  • Both separate business logic from infrastructure
  • Both promote testability and maintainability
  • Both use layered architecture

Differences:

  • Clean Architecture: More abstract, focuses on use cases and entities
  • Onion Architecture: More explicit about domain services and application services
  • Clean Architecture: Uses "Interface Adapters" terminology
  • Onion Architecture: Uses "Infrastructure" and "Application Services" terminology

Practical Example: CQRS with MediatR

Let's explore a practical implementation using Command Query Responsibility Segregation (CQRS) pattern with MediatR, following Clean Architecture principles.

Domain Layer (Core)

// Domain/Entities/User.cs
namespace Domain.Entities
{
    public class User
    {
        public int Id { get; set; }
        public string Name { get; set; }
        public string Email { get; set; }
        public DateTime CreatedAt { get; set; }
        
        // Business rules
        public bool IsValid()
        {
            return !string.IsNullOrEmpty(Email) && 
                   Email.Contains("@");
        }
    }
}

// Domain/Interfaces/IUserRepository.cs
namespace Domain.Interfaces
{
    public interface IUserRepository
    {
        Task<User> GetByIdAsync(int id);
        Task<User> CreateAsync(User user);
        Task<IEnumerable<User>> GetAllAsync();
    }
}

Application Layer (Use Cases)

// Application/Commands/CreateUserCommand.cs
using MediatR;

namespace Application.Commands
{
    public class CreateUserCommand : IRequest<int>
    {
        public string Name { get; set; }
        public string Email { get; set; }
    }
}

// Application/Handlers/CreateUserCommandHandler.cs
using MediatR;
using Domain.Entities;
using Domain.Interfaces;

namespace Application.Handlers
{
    public class CreateUserCommandHandler 
        : IRequestHandler<CreateUserCommand, int>
    {
        private readonly IUserRepository _repository;
        
        public CreateUserCommandHandler(IUserRepository repository)
        {
            _repository = repository;
        }
        
        public async Task<int> Handle(
            CreateUserCommand request, 
            CancellationToken cancellationToken)
        {
            var user = new User
            {
                Name = request.Name,
                Email = request.Email,
                CreatedAt = DateTime.UtcNow
            };
            
            if (!user.IsValid())
                throw new InvalidOperationException("Invalid user data");
            
            return await _repository.CreateAsync(user);
        }
    }
}

// Application/Queries/GetUserQuery.cs
using MediatR;
using Application.DTOs;

namespace Application.Queries
{
    public class GetUserQuery : IRequest<UserDto>
    {
        public int Id { get; set; }
    }
}

// Application/Handlers/GetUserQueryHandler.cs
using MediatR;
using Domain.Interfaces;
using Application.DTOs;

namespace Application.Handlers
{
    public class GetUserQueryHandler 
        : IRequestHandler<GetUserQuery, UserDto>
    {
        private readonly IUserRepository _repository;
        
        public GetUserQueryHandler(IUserRepository repository)
        {
            _repository = repository;
        }
        
        public async Task<UserDto> Handle(
            GetUserQuery request, 
            CancellationToken cancellationToken)
        {
            var user = await _repository.GetByIdAsync(request.Id);
            
            if (user == null)
                return null;
            
            return new UserDto
            {
                Id = user.Id,
                Name = user.Name,
                Email = user.Email,
                CreatedAt = user.CreatedAt
            };
        }
    }
}

Infrastructure Layer

// Infrastructure/Data/UserRepository.cs
using Domain.Entities;
using Domain.Interfaces;
using Infrastructure.Persistence;

namespace Infrastructure.Data
{
    public class UserRepository : IUserRepository
    {
        private readonly ApplicationDbContext _context;
        
        public UserRepository(ApplicationDbContext context)
        {
            _context = context;
        }
        
        public async Task<User> GetByIdAsync(int id)
        {
            return await _context.Users.FindAsync(id);
        }
        
        public async Task<User> CreateAsync(User user)
        {
            _context.Users.Add(user);
            await _context.SaveChangesAsync();
            return user;
        }
        
        public async Task<IEnumerable<User>> GetAllAsync()
        {
            return await _context.Users.ToListAsync();
        }
    }
}

Presentation Layer (API)

// API/Controllers/UsersController.cs
using MediatR;
using Microsoft.AspNetCore.Mvc;
using Application.Commands;
using Application.Queries;

namespace API.Controllers
{
    [ApiController]
    [Route("api/[controller]")]
    public class UsersController : ControllerBase
    {
        private readonly IMediator _mediator;
        
        public UsersController(IMediator mediator)
        {
            _mediator = mediator;
        }
        
        [HttpGet("{id}")]
        public async Task<IActionResult> GetUser(int id)
        {
            var query = new GetUserQuery { Id = id };
            var user = await _mediator.Send(query);
            
            if (user == null)
                return NotFound();
            
            return Ok(user);
        }
        
        [HttpPost]
        public async Task<IActionResult> CreateUser(
            [FromBody] CreateUserCommand command)
        {
            var userId = await _mediator.Send(command);
            return CreatedAtAction(
                nameof(GetUser), 
                new { id = userId }, 
                null);
        }
    }
}

Benefits of This Approach

  • Separation of Concerns: Each layer has a clear responsibility
  • Testability: Business logic can be tested without database or HTTP
  • Flexibility: Easy to swap implementations (e.g., change database)
  • CQRS: Commands and queries are separated, allowing optimization
  • MediatR: Decouples controllers from business logic
  • Maintainability: Changes in one layer don't affect others

Best Practices

1. Dependency Rule

Always ensure dependencies point inward. Outer layers depend on inner layers, never the reverse.

2. Interface Segregation

Define interfaces in the domain layer, implement them in the infrastructure layer.

3. Use Dependency Injection

Inject dependencies through constructors to maintain loose coupling.

4. Keep Domain Pure

The domain layer should have no dependencies on external libraries or frameworks.

5. Use DTOs for Data Transfer

Don't expose domain entities directly to the presentation layer. Use DTOs instead.

6. Implement Repository Pattern

Abstract data access behind repositories to make the domain independent of persistence.
